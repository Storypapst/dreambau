# 🎥 Cap - Open Source Screen Recording

**Cap** is an open source alternative to Loom for beautiful, shareable screen recordings and video messaging.

**Access:** https://cap.wcrbusiness.online

---

## 📋 Overview

Cap Web is a Next.js-based application that allows users to:
- 🎬 Record and share screen recordings
- 💬 Create video messages
- 🔗 Generate shareable links
- 📊 View recording analytics

### Technology Stack

- **Frontend:** Next.js 15, React, TypeScript
- **Backend:** Node.js, Next.js API routes
- **Database:** MySQL (shared with other WCR services)
- **Storage:** Local persistent storage (20GB)
- **Infrastructure:** Kubernetes (k3s)
- **Ingress:** Traefik with automatic HTTPS

---

## 🚀 Quick Start

### Prerequisites

1. **k3s cluster running** with Traefik and cert-manager
2. **MySQL database** available in cluster
3. **Docker** installed for building images
4. **kubectl** access to the cluster

### Deployment Steps

```bash
# 1. Initialize database
sudo ./Dreambau-Cap/scripts/init-database.sh

# 2. Build Docker image (first time or after updates)
cd Dreambau-Cap/docker
sudo ./build.sh

# 3. Import image to k3s
sudo ./import-to-k3s.sh

# 4. Deploy to Kubernetes
cd ../scripts
sudo ./deploy-cap.sh

# 5. Access Cap
# Wait 2-5 minutes for SSL certificate provisioning
# Then open: https://cap.wcrbusiness.online
```

## ⚙️ Configuration

### Database Configuration

Cap uses the shared MySQL instance in the cluster:

```yaml
Database: cap
User: cap
Host: mysql.wcr.svc.cluster.local
Port: 3306
```

Password is configured in `config/cap-secret.yaml`.

### Application Settings

Key environment variables in `cap-secret.yaml`:

| Variable | Description | Example |
|----------|-------------|---------|
| `DATABASE_URL` | MySQL connection string | `mysql://cap:password@host:3306/cap` |
| `NEXT_PUBLIC_URL` | Public URL | `https://cap.wcrbusiness.online` |
| `NEXTAUTH_URL` | NextAuth URL | `https://cap.wcrbusiness.online` |
| `NEXTAUTH_SECRET` | Auth secret | Generated by deploy script |
| `STORAGE_TYPE` | Storage backend | `local` (can be upgraded to S3) |
| `STORAGE_PATH` | Local storage path | `/app/storage` |

### Storage

- **Type:** Kubernetes PersistentVolumeClaim
- **Size:** 20GB (for video recordings)
- **Path:** `/app/storage` in container
- **Mounted from:** `cap-data-pvc`

## 🔧 Common Operations

### View Logs

```bash
# Follow logs in real-time
kubectl logs -f deployment/cap -n wcr

# View last 100 lines
kubectl logs --tail=100 deployment/cap -n wcr

# View logs from last hour
kubectl logs --since=1h deployment/cap -n wcr
```

### Restart Application

```bash
# Graceful restart
sudo kubectl rollout restart deployment/cap -n wcr

# Force restart (delete pods)
sudo kubectl delete pods -n wcr -l app=cap
```

### Shell Access

```bash
# Access container shell
sudo kubectl exec -it deployment/cap -n wcr -- sh

# Run commands
sudo kubectl exec deployment/cap -n wcr -- node -v
```

### Check Status

```bash
# Pods
kubectl get pods -n wcr -l app=cap

# Services
kubectl get svc -n wcr -l app=cap

# Ingress
kubectl get ingress cap-ingress -n wcr

# SSL Certificate
kubectl get certificate cap-wcrbusiness-online-tls -n wcr

# Describe pod (for troubleshooting)
kubectl describe pod -n wcr -l app=cap
```

### Update Application

```bash
# 1. Rebuild Docker image
cd /home/backup/Documents/business/wcr/Dreambau-Cap/docker
sudo ./build.sh

# 2. Import to k3s
sudo ./import-to-k3s.sh

# 3. Restart deployment to use new image
sudo kubectl rollout restart deployment/cap -n wcr

# 4. Monitor rollout
kubectl rollout status deployment/cap -n wcr
```

### Database Operations

```bash
# Access MySQL
kubectl exec -it deployment/mysql -n wcr -- mysql -u cap -p

# Backup database
kubectl exec deployment/mysql -n wcr -- \
  mysqldump -u cap -pCap_Pass_2024_Change_Me cap \
  > cap_backup_$(date +%Y%m%d).sql

# Restore database
cat cap_backup.sql | kubectl exec -i deployment/mysql -n wcr -- \
  mysql -u cap -pCap_Pass_2024_Change_Me cap
```

## 🔍 Troubleshooting

### Cap is not accessible

1. **Check pod status:**
   ```bash
   kubectl get pods -n wcr -l app=cap
   ```

2. **Check logs:**
   ```bash
   kubectl logs deployment/cap -n wcr
   ```

3. **Check ingress:**
   ```bash
   kubectl describe ingress cap-ingress -n wcr
   ```

4. **Check SSL certificate:**
   ```bash
   kubectl describe certificate cap-wcrbusiness-online-tls -n wcr
   ```

### Database connection issues

1. **Verify MySQL is running:**
   ```bash
   kubectl get pods -n wcr -l app=mysql
   ```

2. **Test database connection:**
   ```bash
   kubectl exec deployment/mysql -n wcr -- \
     mysql -u cap -pCap_Pass_2024_Change_Me -e "SELECT 1;" cap
   ```

3. **Check secret:**
   ```bash
   kubectl get secret cap-secret -n wcr -o yaml
   ```

### Pod stuck in CrashLoopBackOff

1. **View logs:**
   ```bash
   kubectl logs deployment/cap -n wcr --previous
   ```

2. **Check events:**
   ```bash
   kubectl describe pod -n wcr -l app=cap
   ```

3. **Common issues:**
   - Database not ready: Wait for MySQL pod
   - Missing migrations: Database should auto-migrate on first run
   - Wrong credentials: Check `cap-secret.yaml`

### SSL certificate not provisioning

1. **Check cert-manager:**
   ```bash
   kubectl get pods -n cert-manager
   ```

2. **Check certificate status:**
   ```bash
   kubectl describe certificate cap-wcrbusiness-online-tls -n wcr
   ```

3. **Check certificate request:**
   ```bash
   kubectl get certificaterequest -n wcr
   ```

4. **View cert-manager logs:**
   ```bash
   kubectl logs -n cert-manager deployment/cert-manager
   ```

## 🔐 Security

### Secret Management

1. **Never commit secrets:**
   - `config/cap-secret.yaml` is gitignored
   - Use `config/cap-secret.yaml.example` as template

2. **Change default passwords:**
   ```bash
   # Database password
   # Update in both cap-secret.yaml and MySQL

   # NEXTAUTH_SECRET
   # Generated automatically by deploy script
   ```

3. **Rotate secrets periodically:**
   ```bash
   # Generate new NEXTAUTH_SECRET
   openssl rand -base64 32
   
   # Update secret
   kubectl edit secret cap-secret -n wcr
   
   # Restart deployment
   kubectl rollout restart deployment/cap -n wcr
   ```

### Network Security

- All external traffic encrypted via HTTPS (Let's Encrypt)
- Internal cluster communication via ClusterIP
- Database only accessible within cluster

## 🔄 Backup & Restore

### Backup

Cap data is backed up by the centralized backup script at `Dreambau-Database/backups/automated-backup.sh`.

**What's backed up:**
1. **Database:** `cap` MySQL database
2. **Files:** `/app/storage` (video recordings)
3. **Configuration:** `cap-secret`

**Backup location:** `/home/backup/Documents/business/wcr/Dreambau-Backup/`

**Manual backup:**
```bash
sudo /home/backup/Documents/business/wcr/Dreambau-Database/backups/automated-backup.sh
```

### Restore

1. **Restore database:**
   ```bash
   BACKUP_FILE="Dreambau-Backup/database/cap_YYYYMMDD_HHMMSS.sql.gz"
   gunzip < $BACKUP_FILE | kubectl exec -i deployment/mysql -n wcr -- \
     mysql -u cap -pCap_Pass_2024_Change_Me cap
   ```

2. **Restore files:**
   ```bash
   kubectl exec -i deployment/cap -n wcr -- \
     tar xzf - -C / < Dreambau-Backup/files/cap_files_YYYYMMDD_HHMMSS.tar.gz
   ```

3. **Restore secrets:**
   ```bash
   kubectl apply -f Dreambau-Backup/config/cap-secret_YYYYMMDD_HHMMSS.yaml
   ```
