# Infisical Human Project Groups Design

## Goal

Make Infisical the authoritative source for human `oriso`, `orimo`, and
`dreambau` access in the Dreambau Testmails registry without granting those
people access to Infisical secret projects.

## Scope

- Create organization groups `testmails-oriso`, `testmails-orimo`, and
  `testmails-dreambau` with organization role `no-access`.
- Do not attach these groups to any Infisical project.
- Add Shazia Kausar to `testmails-oriso` and `testmails-dreambau`.
- Keep Passkey credentials, enrollment codes, human roles, and activation state
  in the local Testmails SQLite database.
- Read project scopes for non-admin Testmails users from Infisical group
  memberships.
- Keep the existing local scopes for Testmails administrators so an Infisical
  outage or accidental group edit cannot lock out the bootstrap administrator.
- Do not create or change AI machine identities, project memberships, secret
  permissions, or secret values in this feature.

## Architecture

Add an `InfisicalHumanAccessProvider` next to the existing registry provider.
It authenticates with the already-mounted Universal Auth credential, reads the
three named organization groups, paginates their existing user memberships,
and maps lower-cased email addresses to Testmails projects. Results are cached
for at most 60 seconds.

The Testmails application synchronizes a non-admin user's project array before
returning human-user data or authorizing account access. A failed Infisical
refresh fails closed with HTTP 503 for non-admin users. An active non-admin who
belongs to none of the three groups remains authenticated but sees no test
accounts.

## Data Flow

1. An administrator changes membership in Infisical.
2. Testmails authenticates to Infisical through Universal Auth.
3. Testmails reads `GET /api/v1/groups` and the member list of each recognized
   group.
4. Group slugs map to `oriso`, `orimo`, and `dreambau`.
5. The resolved array is persisted into the existing `human_users.projects`
   field and used for account scoping.

## Security

- All three groups use `no-access` and are not attached to projects.
- Group membership alone grants no secret access.
- Universal Auth tokens and credentials are never logged or returned.
- Unknown groups are ignored.
- Group reads are bounded and paginated with a maximum page size of 100.
- Non-admin authorization fails closed when membership cannot be refreshed.

## User Experience

The existing Team Access dialog continues showing the effective project list.
Its description states that project access is synchronized from Infisical.
Enrollment codes remain one-time values generated by Testmails.

## Verification

- Unit tests cover group mapping, pagination, caching, unknown groups, and API
  failure.
- Authentication tests cover member synchronization, zero-group access,
  administrator independence, and fail-closed behavior.
- The full lint, unit-test, and build commands pass.
- Live verification proves that Shazia is in the two no-access groups, that the
  groups have no Infisical project memberships, and that Testmails reports her
  effective scopes as `oriso,dreambau` without exposing an enrollment code or
  secret.
